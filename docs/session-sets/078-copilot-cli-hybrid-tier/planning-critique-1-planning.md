M1 — **The weekend is arithmetically impossible.** Set 077 has 4 sessions left (2/6 done) and this set is 4 sessions, gated on 077 completing. That's 8 serialized sessions between 2026-07-02 and 2026-07-06. The escape hatch (parallel worktree, lift the prerequisite) doesn't actually recover the timeline — it just trades a scheduling failure for a merge/release-ordering failure (see M2). Change: drop the weekend deadline from the plan of record, or explicitly scope the *minimum shippable* subset (S1 + S2 only, adapter behind a flag, no release) as the weekend target and move S3–S4 + 0.28.0 past 077's completion.

M2 — **Machine gate and prose contradict; parallel mode breaks release ordering.** `Session Set Configuration` hard-codes `prerequisites: [077, condition: complete]`, but the prose authorizes the operator to "lift the prerequisite" and run in parallel. Tooling will enforce the config and block the parallel plan, so the escape hatch is fiction as written. Worse, 0.28.0 "on top of" 077's 0.27.0 means the release is built on a base that, in parallel mode, doesn't yet exist — you'd publish 0.28.0 without 0.27.0's shipped changes, or have to re-cut it after 077 merges. Change: pick one — either keep the hard gate (and drop the weekend), or define an explicit `prerequisite: partial` mode with a stated rebase-onto-0.27.0 step and a rule that 0.28.0 is not tagged until 077 is released.

M3 — **75% of the set (and the release) is contingent on an unverified external assumption, with no in-timeline fallback.** S1's fail-loud checkpoint can legitimately stop the entire set. The named fallback (GitHub Models API adapter) is explicitly a *future* set, so an S1 stop delivers *nothing* to a team that needs this for a mission-critical week. Building a critical-path commitment on a contract that "predates the knowledge cutoff and corporate builds may differ" is the central risk. Change: run the S1 contract-pin spike *before* committing the weekend/authoring S2–S4 as fixed work; and fold a minimal GitHub Models REST fallback path into scope (or an explicit "if S1 stops, the deliverable is Lightweight tier + findings" statement) so an S1 stop still leaves the team with a plan.

M4 — **The cross-provider guarantee — the whole value prop — is an unproven assumption about the Copilot catalog, not just about headless mode.** The set assumes (a) the `--model` picker exposes each entry's *underlying provider identity*, and (b) at least two provider-distinct models are actually *enabled on the seat*. Enterprise policy can enable only one family (e.g. GPT-only), in which case the verifier picker's cross-provider rule fails loudly and `route()` cannot run at all — even though headless mode and model selection technically "work." The S1 checkpoint only tests headless+picker availability, not this. Change: add to S1's fail-loud criteria an explicit check that ≥2 models resolving to *different* underlying providers are enabled and dispatchable on the operator's seat; if not, stop with the same posture as a missing headless mode.

M5 — **The operator's seat is not evidence about the team's seats.** All live verification (S3 dogfood, S4 UAT) runs on *the operator's* authenticated seat, but Copilot model availability, premium-quota, and headless enablement can vary per user/policy within the same org. A green attestation on the operator's seat does not prove the Copilot-locked team can run the Full tier during their critical week — which is the actual objective. Change: verify (or have the operator verify) the model catalog + headless availability against at least one representative *team* seat in S1, and record the enterprise-policy dependency as an explicit precondition in the docs/UAT.

M6 — **Metrics schema changes with `contractGate: none` and no consumer-compat verification.** S3 adds `transport`, `premium_requests`, `usage_estimated` to `router-metrics.jsonl` rows and changes `cost_report` rendering. Nothing in the plan verifies existing consumers of the metrics file (including anything 077 ships) still parse it, and `contractGate: none` means the mechanical suite won't catch a schema regression. This is a deliverable ("truthful accounting") whose backward-compat nothing verifies. Change: add a metrics-row schema/back-compat test (old readers tolerate new fields; `cost_report` handles absent-cost rows) and state the metrics-schema-additive contract explicitly; reconsider `contractGate: none`.

M7 — **No hard premium-quota ceiling; retries + mandatory verification can silently burn a team's quota during the critical week.** The design reuses the existing retry/backoff loop over subprocess calls, each verified cross-provider (2× calls minimum), and the unknown-cost guard posture is "do not skip verification, do not auto-escalate" with only a *soft* `premium_requests` budget. Under retries against a metered seat, that's an uncapped burn on the exact resource the team depends on that week. Change: add a *hard* `premium_requests` circuit breaker (fail-loud on ceiling, operator-visible) as a first-class S3 deliverable, not a soft analogue; cap retries explicitly for the CLI transport.

M8 — **The "additive surface, avoid 077's files" claim is not bounded against 077's actual diff.** The escape hatch says avoid `close_session/start_session/dedicated_verification/session_state`, but S3 touches `verification.py` (picker), `__init__.py` (route), and `metrics.py` — and 077 is *also* copilot/verification hardening on the same team's Full/Lightweight machinery. Overlap on `verification.py`/`metrics.py` is plausible and unverified. Change: before authorizing parallel mode, enumerate 077 S3–S6's full touched-file set and prove disjointness (or list the concrete conflict points and a merge plan); don't assert additivity abstractly.

M9 — **S1 rewrites S2–S4 scope, so the back half isn't fixed-scope work, and there's no rollback plan for a mid-week bad release.** S1 step 4 amends "Feature scopes where the real contract moved them," meaning S2–S4 are estimates, not commitments — planning them as three fixed 1-session units (with zero slack, on a weekend) is unsound. Separately, the only verification of the live path is a single operator self-attested UAT; if 0.28.0 ships broken into the mission-critical week, there's no named rollback/pin-to-0.27.0 procedure. Change: label S2–S4 as contingent-scope pending S1, add explicit slack, and add a rollback/downgrade step (pin router 0.27.0, transport disabled) to S4's release checklist.

**Verdict: SOUND-WITH-CHANGES** — the transport-not-a-tier architecture is coherent, but the timeline is fantasy and several existential assumptions (S1 stop, cross-provider catalog, quota) need explicit contingencies before this is a plan rather than a hope.
