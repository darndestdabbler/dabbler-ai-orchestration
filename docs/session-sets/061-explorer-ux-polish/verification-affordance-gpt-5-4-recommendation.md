# Verification Affordance Recommendation

1. The affordance should be a quiet marker in the existing marker strip, not a badge and not an inline toggle. Put it immediately after the lightweight marker and before the prerequisite marker, so the cluster reads as: tier, verification, dependency. I would use two compact states only: `v?` for manual-or-unknown verification posture, and `v+` for dedicated verification still owed or still in flight. The marker should be de-emphasized text with a help cursor, matching the Set 061 row-quieting direction in the spec and the existing lightweight marker pattern. Tooltip copy should do the real work:

`Lightweight manual verification. The Explorer cannot know whether this set was reviewed out of band. Click for verification options.`

`Lightweight dedicated verification is enabled. A verification or remediation session is still owed or in flight. Click for the next step.`

2. Do not try to surface every underlying workflow state on the row. Collapse the derived machine states into three visual outcomes. First, manual-or-unknown: `verificationMode` is `out-of-band-or-none`. This is not "unverified"; it is "the system cannot attest." Show `v?` only when the set is not started and still switchable, or when work is complete and the operator now needs the manual review path. Second, dedicated pending: `verificationMode` is `dedicated-sessions` and the set is in `awaiting-verification`, `awaiting-remediation`, or `awaiting-human`. Show `v+` and vary only the tooltip and click action. Third, dedicated closed: `closed-verified` or `closed-dispositioned`. Show no verification marker at all; quiet is the success state. I would not attempt a positive "verified" badge, and I would not claim Mode A is unverified, because with spec plus ledger alone you cannot know that honestly.

3. The affordance should not flip Mode A to Mode B except on not-started sets, and even there it should be a secondary action, not the default click. The current contract is that the choice is captured once at set start in the workflow, and the extension already treats adjacent semantics that way for tier switching. So the only legal mutation is: on a not-started lightweight Mode A row, the marker's action sheet may offer `Enable dedicated verification sessions...`, which rewrites `spec.md` from `out-of-band-or-none` to `dedicated-sessions` and refreshes. It should not append a typed session, and it should not silently do anything else. On any started or completed set, do not offer the toggle. The realistic "I finished the work, now I want dedicated verification" case is not a toggle problem; it is a new workflow exception, and the UI should not fake one.

4. The lowest-friction path is a state-specific copied prompt, because that is already how the extension hands work to agents. For Mode A complete rows, clicking `v?` should copy the existing set-review prompt and immediately offer `Open external verification note`, reusing the existing `external-verification.md` command. For Mode B pending rows, the extension needs one new command: `Copy verification kickoff prompt.` That prompt should tell the agent to run the dedicated verification flow on a different engine, using the typed-session workflow. I would not make `open the doc` the primary path, and I would not rely on the generic `Start Next Session` copy prompt here, because the dedicated flow is not generic: it explicitly requires a typed verification session and different-engine handoff.

5. Ship this as two pieces, not one monolith. Piece one is the useful part: render the verification marker, map it to honest derived states, and make click copy the right next-step prompt. Piece two is optional and smaller: the not-started-only spec rewrite that enables dedicated verification. The biggest risks are misleading users on Mode A, implying lightweight verification is automatic, and encouraging an illegal mid-set mode flip. Those are all avoidable if the UI says "manual or unknown" instead of "unverified," uses copy-prompt language instead of `verify now`, and never offers A-to-B mutation after the first session has started.

Bottom line: add a quiet, clickable verification marker, but make it an action handoff surface, not a general toggle. Use `v?` for manual-or-unknown Mode A and `v+` for dedicated verification owed or active; drop the marker once the dedicated path closes cleanly. Ship the prompt-copy handoff first, and if you add mutation at all, keep it strictly not-started-only and secondary.
