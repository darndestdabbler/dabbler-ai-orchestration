# gemini (gemini-2.5-pro)

**Q1. Is the recurring context cost a real problem?**
Yes, it's a critical problem, not premature optimization. The cost is threefold:
1.  **Monetary:** 50k tokens is ~$0.25/session on GPT-4-Turbo input. At 40 sessions/day, that's $10/day or $3,650/year of pure, recurring overhead.
2.  **Performance:** It's "attention" cost. The model wastes cycles parsing irrelevant or low-value instructions, increasing the risk of missing the critical ones. This is the "needle in a haystack" problem.
3.  **Opportunity:** Those 50k tokens could be used for more relevant context: longer code files, API docs, or richer user prompts.

The tradeoff is currently unbalanced. You're paying a high, constant tax on *every* session to prevent a smaller number of potential mistakes.

**Q2. Is a hard size budget the right trigger?**
No. It's a crude proxy for value. It forces a choice between a new, critical lesson and an old, still-valuable one based on an arbitrary byte count. The best triggers are signals of obsolescence or assimilation, not size. A better trigger is when a lesson is **encoded into automation**—e.g., a new linter rule, a static analysis check, or a reusable library function now enforces the lesson, making the text redundant.

**Q3. Is the "promote within 10 sets or archive" rule good or flawed?**
Deeply flawed. It incorrectly conflates frequency with value. A critical lesson for a rare but catastrophic failure mode (e.g., "how to recover from database migration corruption") might only be relevant once every 50 sessions. This rule would wrongly evict your most valuable disaster-recovery knowledge. It punishes durable tactics that are not strategic "principles."

**Q4. What is the BEST signal that a lesson should be archived?**
The best signal is a combination of **disuse and operator judgment**. A concrete, checkable criterion: **A lesson is a candidate for archival if it has not been explicitly cited as "instrumental" in a session-set summary for the last `N` (e.g., 20) session sets.** This requires an explicit feedback loop instead of passive time decay.

**Q5. Recommend a concrete mechanism.**
Implement a "citation-based relevancy" system.

*   **1. ID Everything:** Assign a short, stable, unique ID to every lesson and guidance point (e.g., `L-101`, `G-25`).
*   **2. Mandatory Citation:** The session-set summary, a required artifact, MUST list the IDs of all lessons/guidance that were instrumental to that session's success.
*   **3. Track Usage:** A simple script parses these summaries and updates a metadata file (`usage.json`) tracking the last-cited session number for each ID.
*   **4. Trigger Archival Review:** A bot runs every 10 session sets. It identifies all lessons not cited in the last 20 sets and proposes them for archival in a pull request.
*   **5. Operator-in-the-Loop:** The operator reviews the PR. They can approve the archival, or "touch" a lesson (updating its last-cited metadata) with a comment like `/keep L-101 reason: still critical for edge cases`.
*   **6. Archive, Don't Delete:** "Archiving" moves the lesson text and its ID to `lessons-archive.md`. This file is NOT loaded into context but is preserved.
*   **7. On-Demand Retrieval:** Index the archive in a vector database. The agent can be given a tool to perform a similarity search against the archive if it encounters a novel problem, retrieving old lessons only when potentially relevant.

---
**VERDICT:** The operator's proposal is **partly right** in identifying the problem but its solutions are wrong; replace static size/age rules with a dynamic feedback loop that tracks explicit lesson application.
