ISSUES FOUND

- **Issue 1: Round 5’s L14 remediation permanently exempts accepted fixes even when later remediation regresses them**
  - **Category:** Correctness
  - **Severity:** Major
  - **Failure scenario:** A first remediation-review accepts several fixes but rejects one, causing the normal second remediation cycle. Staff using the weaker deployment models edits the same fix area and accidentally removes or damages an already-accepted fix—a probable scenario given the supplied evidence that roughly one-third of later findings were remediation-induced. The second review is told not to re-verdict that finding, and if the edit restores the affected code to its discovery-baseline state, the cumulative fix delta contains no hunk for it. The review can therefore return `VERIFIED` and close with the original Major reintroduced.
  - **Details:**
    - **Violation:** This specifically challenges the Round 5 L14 remediation. It claims that a previously accepted ID can permanently become “**EXEMPT from re-coverage**,” while the deliverable’s objective is to catch defects introduced by subsequent remediation. No-resurrection applies to fresh wording over unchanged evidence; it cannot suppress a defect reintroduced by later edits.
    - **Impact:** A normal two-cycle remediation path can finish successfully with an original Critical/Major defect present again. That defeats both per-finding settlement and fix-induced-defect review, materially changing the merge decision.
    - **Evidence:** `assemble_cross_round_ledger_with_ids()` unions all prior `fix-accepted` IDs into `accepted_ids` and removes them from `required_ids` without checking whether the working tree changed after acceptance. `build_phase_framing()` explicitly says accepted IDs are exempt and “do not re-verdict them.” Meanwhile, `assemble_fix_delta_evidence()` supplies only the cumulative `discovery baseline -> current tree` diff and stores no prior-review tree. If a later remediation removes an accepted addition and returns that region to its discovery state, the hunk disappears from the evidence entirely. The duplicate-chain coverage check also treats every exempt ID as terminal regardless of subsequent edits.
    - **Fix:** Invalidate prior acceptance after subsequent remediation unless the accepted fix is proven unchanged. At minimum, require re-verdicting all previously accepted IDs on another remediation cycle. A narrower implementation can persist each review’s tree and include the since-last-review delta, then re-open accepted IDs whose fixing paths/hunks were changed or reverted.
