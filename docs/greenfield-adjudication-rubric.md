### The Adjudication Procedure

1.  **Work from the consolidated report ONLY.** Adjudication is performed on the deduped, unioned `remediation-report.md` generated by the matrix run. Do not inspect raw provider-branded outputs.
2.  **Apply the anti-bias rule.** The adjudicator's task is to evaluate the substance of each finding, independent of its source. Ignoring provider provenance during the TP/FP call itself prevents brand bias (e.g., favoring a high-status provider) from corrupting the verdict. The provenance is used later for scoring, not for the initial judgment.

### The Four Verdicts

Assign exactly one verdict to each consolidated finding.

#### TP (True Positive)

-   **Decision Rule:** The finding identifies a real defect or an actionable problem in the diff's code or the repo's state. The adjudicator must be able to articulate the concrete problem and a plausible path to it causing harm or incorrect behavior. Severity (Critical/Major/Minor) is a separate field recorded alongside the TP verdict.
-   **Example:** A finding points out that a new function lacks input validation for a `user_id` parameter, which could lead to an injection vulnerability if a downstream consumer passes unsanitized input. This is a TP.

#### FP (False Positive)

-   **Decision Rule:** The finding is incorrect, irrelevant, or unactionable. It does not identify a real defect. This includes misinterpretations, complaints about correct code, hallucinations, or pure style nits with no plausible path to harm.
-   **Example (Push-Elision):** A `push` surface finding complains that a function is incomplete or has a syntax error, but the error is an artifact of the diff context being elided (`...`).
-   **Example (Pull-Out-of-Sandbox):** A `pull` surface finding correctly identifies a defect in an imported library (`from some_lib import broken_func`) but incorrectly assigns blame to the diff's code, which is merely calling the library correctly. The defect is real but exists outside the review sandbox.

#### duplicate

-   **Decision Rule:** The finding substantively restates a defect already described by another consolidated finding that will be marked TP. This occurs when the consolidation logic fails to merge two findings with different wording or a missing stable key. A `duplicate` verdict is not scored as a TP or FP; it is effectively folded into the finding it duplicates.
-   **Example:** Finding A: "Variable `x` is used before assignment on line 52." Finding B: "Uninitialized variable `x` on L52 can cause a runtime error." If these are separate consolidated findings, one is marked TP, the other `duplicate`.

#### unclear

-   **Decision Rule:** The adjudicator cannot confidently determine TP vs. FP from the provided report and context. The claim is ambiguous, requires running code to verify, or lacks sufficient evidence. Per operator default, `unclear` findings are flagged for the synthesis set and are **excluded** from precision calculations (they are neither TP nor FP). Do not silently default an `unclear` finding to FP.
-   **Example:** A finding claims "this logic may lead to a race condition under high load" without providing a specific code path or interleaving that demonstrates the race. Verifying this would require a complex setup beyond the scope of adjudication.

### The Adjudicated Union (Proxy Denominator)

The **adjudicated union** is the set of all unique, distinct findings that were marked **TP** during adjudication, regardless of their source (i.e., from any matrix arm or from the session's human verification).

This set serves as the proxy denominator for calculating relative performance metrics like "share-of-union."

**Crucially, this is NOT recall.** The adjudicated union can only contain defects that at least one source surfaced. It cannot account for unknown-unknowns (defects that no tool and no human found).

### The Scoring Unit: the provider×surface arm

The scoring unit is the **provider×surface arm** — `push:anthropic`, `pull:openai`,
`pull:google` — NOT the `(push, pull)` matrix **cell**. Sets 072–073 use "cell" for a
single dual-surface `(push, pull)` run; the established 2-cell matrix
(`push:anthropic` paired with each of `pull:openai` / `pull:google`) is two such runs.
(Note `push:anthropic` runs once per cell — in Set 073 the two push runs returned
different verdicts; its TP set is the union across cells.)

**Two reports, two jobs — this is the part that makes per-arm scoring reproducible:**

- The **consolidated `remediation-report.md`** drives the **verdict**: it is the
  deduped, **provider-blind** finding list, and its provenance is only **surface-level**
  (`push-only` / `pull-only` / `both`). Adjudicate TP/FP/dup/unclear from it (the
  anti-bias rule). Surface-level provenance **cannot** tell `pull:openai` from
  `pull:google`, so it is *not* sufficient for per-arm attribution on its own.
- The per-cell **`verification-matrix-report.json`** drives the **arm provenance**: each
  cell names its push and pull provider, so a pull finding in the `(push:anthropic,
  pull:openai)` cell is attributed to `pull:openai`, and one in the `(push:anthropic,
  pull:google)` cell to `pull:google`. Reading off which arm(s) caught a finding is a
  **mechanical lookup**, not a judgment, so consulting the provider-identified matrix
  report here does **not** reintroduce adjudication bias (the verdict was already fixed
  provider-blind above).

So the adjudicator records, for each TP, the explicit `armsCaught` list (§ Verdict
Recording) by matching the consolidated finding back to the matrix report's per-cell
findings. The per-arm TP/FP/unique-TP counts are computed from those `armsCaught`
lists, never from the surface-level `push-only/pull-only/both` tag alone.

### Per-Arm Scoring Table

After adjudication, a scoring table is generated for each provider×surface arm:

| arm (surface:provider) | TP | FP | precision        | share-of-union | unique-TPs | arm_cost_usd | cost-per-TP   |
| ---------------------- | -- | -- | ---------------- | -------------- | ---------- | ------------ | ------------- |
| `push:anthropic`       | 8  | 2  | 0.80             | 0.67           | 3          | $0.45        | $0.056        |
| `pull:openai`          | 5  | 0  | 1.00             | 0.42           | 1          | $0.15        | $0.030        |
| `pull:google`          | 0  | 0  | n/a (0 findings) | 0.00           | 0          | $0.18        | $0.18 (0 TP)  |

*(Illustrative numbers; `|adjudicated_union_TP|` = 12 in this example. The
`pull:google` row demonstrates the two edge cases below — a clean-verdict arm.)*

**Column Formulae:** (all per-arm counts come from the `armsCaught` lists, derived as
above — never from the consolidated report's surface-level `push-only/pull-only/both` tag.)
-   `TP`, `FP`: counts of findings whose `armsCaught` includes this arm with the corresponding verdict.
-   `precision`: `TP / (TP + FP)`.
-   `share-of-union`: `arm_TP / |adjudicated_union_TP|`.
-   `unique-TPs`: count of TPs whose `armsCaught` is *this arm only*.
-   `cost-per-TP`: `arm_cost_usd / TP`.

**Edge Cases:**
-   **TP=0:** report `cost-per-TP` as the arm's total cost followed by "(0 TP)", never infinity or an error.
-   **TP+FP=0:** a clean-verdict arm (no findings) has undefined `precision`. Report "n/a (0 findings)", not 1.0 or 0.0.

**Worked Flow:**
A consolidated finding `F1` (a null-pointer risk) is marked `TP` from the
provider-blind remediation report; matching it back to the matrix report shows it in
the `(push:anthropic, pull:openai)` cell on both surfaces, so `armsCaught:
[push:anthropic, pull:openai]`. Result: `F1` increments the
`TP` count for **both** of those arms, contributes 1 to `|adjudicated_union_TP|`, and
counts as a `unique-TP` for **neither** (two arms caught it).

### Verdict Recording

Per-finding verdicts and the provenance mapping (which **arm(s)** get credit for each
TP) are recorded in the session's `matrix-run/adjudication.md` file. Record each
finding with an explicit arm list, e.g.:

```md
- F1 — null-pointer risk — verdict: TP (Major) — armsCaught: [push:anthropic, pull:openai]
```

This file is the canonical record of the adjudication decisions and is included in
the telemetry package shipped from the consumer repo. The directory structure is
defined by the D5 protocol in `docs/greenfield-matrix-protocol.md`.

### Stratify by Diff Class

All scoring is stratified by diff class (e.g., `source-dominated`, `packaging-small`). Results are never pooled across classes, as this would violate the experimental design (see Validity Threat 1). The `dabbler-platform` (source-dominated) results are the primary signal. For the full protocol on diff classes and validity threats, see `docs/greenfield-matrix-protocol.md`.
