- **Issue** → The webview seed is applied only once globally, so state from one repo/root can bleed into another live repo/root. In a multi-root session or any root switch while the same webview stays alive, the second root never gets its own durable `tierSeed`; the old root’s `tier`, `budget`, `zeroMethod`, `parallel`, and warning state keep driving the form.  
  **Location** → `tools/dabbler-ai-orchestration/media/session-sets-tree/client.js` (`const gsState = ...`, `let tierSeedApplied = false`, and the first-snapshot seed block).  
  **Fix** → Scope persisted form state to the current repo root/workspace identity. Persist that identity alongside `gsState`, reset/reseed when it changes, and replace the single `tierSeedApplied` boolean with per-root seed application.

- **Issue** → `tierDirty` is a sticky bit with no reset path. Once a user flips the radio, that persisted `true` survives indefinitely, so later sanctioned marker changes can be ignored even after the durable tier has caught up. That violates the intended “flipped after the last seed” semantics and can leave the form/prompt on a stale tier.  
  **Location** → `tools/dabbler-ai-orchestration/media/session-sets-tree/client.js` (radio `change` handler and persisted `tierDirty`) and `tools/dabbler-ai-orchestration/media/session-sets-tree/gettingStartedHtml.js` (`restoreGsState` precedence logic).  
  **Fix** → Track the last applied seed/root in persisted state and clear `tierDirty` when a valid seed is applied or already matches the current tier. At minimum, when an incoming durable seed equals `gsState.tier`, persist `tierDirty: false` so only unresolved post-seed flips stay dirty.
