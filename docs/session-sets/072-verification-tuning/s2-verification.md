**ISSUES_FOUND**

- **Major** Issue -> `validate_matrix_report` does not reject semantically bad cells, violating L-066-1 parity and the "bad-cell" requirement. The producer always emits a dual-surface cell with shared confounds echoed into every row (`_cell_telemetry()` hardcodes `surfaces=(push, pull)` and copies one `orchestrator`/`diff_shape` into every cell), but `_validate_telemetry()` accepts any non-empty subset of surfaces and `validate_matrix_report()` never checks a cell's `orchestrator*`/`diff*` values against the top-level `orchestrator`/`diffShape`. Impact -> malformed artifacts such as single-surface cells or per-cell confound drift still validate, so downstream analysis can silently mix incomparable data even though this mode guarantees one dual-surface run over one shared diff. Location -> `ai_router/verification_only_app.py::_cell_telemetry`, `_validate_telemetry`, `validate_matrix_report`. Fix -> require `surfaces` to be exactly `["push", "pull"]` (no missing/duplicate entries) and, during `validate_matrix_report()`, enforce equality between each cell's echoed `orchestrator*`/`diff*` fields and the top-level `orchestrator`/`diffShape`.

- **Major** Issue -> the report validators do not mirror the producer's provenance invariants for `provenanceComplete`, `pushUnkeyed`, and `pullUnkeyed`, violating L-066-1. `CellResult.to_dict()` and `build_remediation_report()` serialize these fields directly from `merge_findings`, but `_validate_cell()` only type-checks them, and `validate_remediation_report()` only rejects the one-way case where `provenanceComplete` is `true` with unkeyed data. It never enforces the converse or exact count consistency. Impact -> malformed reports can validate with arbitrary unkeyed counters or `provenanceComplete: false` despite fully keyed findings, so consumers cannot trust whether the artifact is fully merged vs safely over-split. Location -> `ai_router/verification_only_app.py::_validate_cell`, `validate_matrix_report`, `validate_remediation_report`. Fix -> recompute expected unkeyed counts from `findings` and require exact matches, plus bidirectional consistency: `provenanceComplete` iff both unkeyed counts are zero and every finding is keyed.
