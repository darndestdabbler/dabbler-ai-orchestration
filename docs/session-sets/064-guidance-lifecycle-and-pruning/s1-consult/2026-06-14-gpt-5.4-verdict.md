# gpt (gpt-5.4)

Q1. **Yes, real problem, not premature optimization.**  
50k tokens/session is expensive in both **money, latency, and attention dilution**.

- At 69 session sets, you are paying that cost **every single start**, whether 5% or 80% of the content matters.
- 50k input tokens is often a **material fraction of the useful reasoning budget** for the actual task.
- The hidden cost is worse than API spend: long evergreen docs become **low-signal wallpaper**, so genuinely important rules lose salience.

Rule of thumb: if these files prevent even one serious repeat mistake every few sessions, they are worth having. But **50k/session for monotonic append-only memory is too blunt**. Manage it.

---

Q2. **Hard size budget is useful as a backstop, but not the main trigger.**  
A pure size gate creates bad behavior: people optimize for bytes, not usefulness.

Better primary triggers:

1. **Superseded** — replaced by a better/generalized lesson.
2. **Encoded elsewhere** — now enforced by lint/test/checklist/tooling/template.
3. **Unused** — not cited, not promoted, not implicated in reviews/incidents for N session sets.
4. **Low applicability** — only relevant to a retired subsystem or one-off migration.

So: keep a **size ceiling**, but archive based on **earning its keep**, not just file length.

Q3. **Flawed.**  
“Promote within 10 sets or archive” assumes promotion is the only proof of value. It isn’t.

Your example is exactly the failure mode: durable tactical guidance often should **never** become Principle/Convention, yet remains worth keeping:
- “halve the batch and retry on truncation”
- “when codegen diff is huge, split by file”
- “reproduce with minimal fixture before patching parser”

Those are recurring operator tactics, not project doctrine. A 10-set timeout would wrongly evict them.  
Promotion should mean “broad normative rule,” not “survived long enough.”

---

Q4. **Best signal: no demonstrated reuse and not encoded elsewhere.**  
Concrete criterion:

> Archive a lesson if, over the last **N=12 session sets**, it has **zero evidence of use** and is **not referenced by any active guidance, checklist, test, linter, template, or incident writeup**.

Make “evidence of use” checkable:
- explicitly cited in session summary / plan / retro
- linked from project-guidance
- triggered by a review checklist
- tied to a test/automation added because of it
- mentioned in incident/postmortem

If none of those happened for 12 sets, it’s not earning startup-context cost.

Q5. **Mechanism I would actually implement**
- **Split memory into 3 tiers:** `project-guidance.md` (always load), `active-lessons.md` (always load, capped), `archive-lessons.md` (never auto-load).
- **Add metadata per lesson:** `added-set`, `last-used-set`, `status`, `superseded-by`, `encoded-in`, `scope`. Minimal YAML or inline tags is enough.
- **Require explicit “used this lesson” marking** in the session wrap-up when a lesson influenced work; update `last-used-set`.
- **Archive trigger runs at end of each session set** via curator review or script, not ad hoc during writing.
- **Archive if any of these are true:** superseded, encoded into automation, subsystem retired, or `last-used-set` older than 12 sets.
- **Cap active lessons by count/tokens**, not raw total repo memory; e.g. keep `active-lessons` under ~10–15k tokens.
- **Promotion is orthogonal:** promote only when something becomes a stable project-wide rule; do not use promotion failure as archive trigger.
- **Archive ≠ delete:** move full text + metadata + rationale to `archive-lessons.md` (or `/memory/archive/...`), leave a one-line pointer if still historically relevant.

**VERDICT:** partly right — the cost is real and needs control, but the biggest change is: **replace the “promote within 10 sets or archive” rule with an evidence-of-reuse/encoded-elsewhere rule.**
